# Block V7 — Project context

This document summarizes what Block is, how the repo is structured, and key concepts so that context doesn’t need to be repeated in every conversation.

---

## What Block is

**Block V7** is a multi-tenant, invite-only **field sales and operations platform** for door-to-door and field teams. It provides:

- **Territory management** — Cluster sets (generated by algorithm or drawn as zones), clusters, assignment to reps. Managers can set optional schedule (scheduled_start / scheduled_end) per cluster on the territory detail page; schedule is shown in the table and inspector and on the Knocks page. The Schedule page shows labor jobs and scheduled clusters.
- **Sales pipeline** — Leads, quotes, sales, contracts, follow-ups.
- **Rep and labor management** — Reps (sales), laborers (field techs), jobs, availability, time-off.
- **Messaging** — Threads, Twilio SMS integration, webhooks.
- **Maps** — Mapbox on web for territories, cluster inspection, drawn zones.
- **Analytics and audit** — Dashboard, analytics, audit log.
- **Payments** — Stripe (checkout, intents, webhooks).
- **Exports** — CSV/JSON exports (sales, assignments) via async worker.

## Architecture

- **Web** never talks to Supabase directly. It calls the API (same-origin proxy in production so cookies work).
- **API** is the only backend that uses Supabase (Postgres + Storage). Uses service role for worker and privileged routes.
- **Worker** runs in the same process as the API (Railway Procfile: `web` + `worker`). Polls `jobs_queue`, processes jobs, updates DB.
- **Auth:** Web uses **cookie-based sessions** (CSRF required for state-changing requests). Mobile uses **bearer tokens**. Login/refresh and webhooks skip CSRF.

## Key concepts

- **Organization (org)** — Tenant. All data is scoped by `org_id`. Counties, properties, cluster sets, reps, laborers, sales belong to an org.
- **County** — Geographic area per org. Properties and cluster sets (worker-generated) are tied to a county. Stored in `counties` with `org_id`.
- **Cluster set** — A collection of clusters. Either **worker-generated** (created via “Create a cluster set” with county + filters; `county_id` set) or **drawn zones** (created from Draw zone; `county_id` null). Worker only runs for worker-generated sets.
- **Cluster** — A group of properties (polygon on map). Has `hull_geojson`, center, `stats_json` (size, value), assignment to rep, and optional `scheduled_start` / `scheduled_end` (ISO) for rep scheduling. When null, rep app shows Unscheduled and omits from Calendar. Belongs to one cluster set.
- **Properties** — Stored in `properties` with `org_id`, `county_id`, lat/lng, address, value_estimate. Cluster generation and draw-zone backfill query properties by org (and county for worker).
- **Rep** — Sales rep. Has `reps` row (org, profile, name, home_lat/lng). Assigned to clusters. Location tracked via `/v1/reps/me/location`.
- **Laborer** — Field tech. Has `laborers` row. Has jobs, availability, time-off. Used by labor mobile app.
- **DNK** — Do Not Knock. Outcome in interactions; “Exclude DNK” in cluster generation filters out properties with prior DNK outcome.
- **Interactions** — Knock/visit outcomes per property (e.g. not_home, lead, do_not_knock). Used for “only unworked” and “exclude DNK” filters.

## Cluster generation (worker)

1. User creates a cluster set on Territories (county + radius, min houses, value range, exclude DNK, only unworked).
2. API inserts `cluster_sets` row (status `queued`) and a `jobs_queue` row with type `cluster_generate`.
3. Worker picks up the job, loads properties for that org/county (with value and interaction filters), runs **DBScan** (`dbscanCluster` from shared) with `radius_m` and `min_houses`, then writes `clusters` and `cluster_properties`. No hardcoded limit on cluster count; only listing is capped (default 2000, max 5000).
4. Smaller radius and lower min houses → more, smaller clusters; larger radius and higher min houses → fewer, larger clusters.

---

## Draw zones

- User draws one or more polygons on the Draw zone page; submits as one cluster set.
- `POST /v1/zones` with `zones: GeoJSON.Polygon[]` creates one cluster set (county_id null) and one cluster per polygon, with optional property backfill (point-in-polygon) and `cluster_properties` + stats. No worker job.

## Auth and access

- **Web:** Cookie session. CSRF header required for POST/PUT/PATCH/DELETE except login, refresh, invite accept, and webhooks.
- **Mobile:** Bearer token; no CSRF.
- **Roles:** `admin`, `manager`, `rep`, `labor`. Admin/manager use web; rep/labor use mobile. Invites can create any role; accept flow creates profile and optionally `reps` or `laborers` row.

---

## Env and deployment

- **API:** `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `COOKIE_DOMAIN`, `WEB_BASE_URL`, etc. See README and deployment docs.
- **Web:** `NEXT_PUBLIC_API_URL`, `NEXT_PUBLIC_MAPBOX_TOKEN`, etc.
- **Railway:** Procfile runs both `web` (Fastify) and `worker` (Node processor). Both use same env.

---

## Conventions

- API route handlers use `requireManager`, `requireAnyAuthed`, or role checks from `routes/_helpers.js`.
- Supabase client in API/worker is from `createServiceClient()` (service role).
- Web uses `api` helper from `lib/api.ts`; throws `ApiError` with `status` and `data` for non-2xx.
- Cluster set list default limit 2000, max 5000; territory detail page requests `limit=5000` so all clusters for a set are shown.
